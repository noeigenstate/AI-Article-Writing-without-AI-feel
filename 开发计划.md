# Speak Plainly — 去 AI 味文章改写工具 · 开发计划

> 目标：上传一篇 Word 文章 + 几篇风格相近的范文，让本地 AI 模仿范文风格改写正文、去除"AI 味"；
> 改写后逐句对照，用户可对任意句子**点击即时获取多个表达选项**，也可手动编辑，最终导出新的 Word。

---

## 1. 已确认的设计基线

| 决策项 | 选择 | 影响 |
|--------|------|------|
| 文章语言 | 主要中文 | 切句按中文标点；改写 prompt 针对中文 AI 味 |
| 选项生成时机 | **点击时即时生成** | 先整篇改写一次，单句候选按需生成 → 省算力、体验好 |
| AI 后端 | **Xiaomi MiMo v2.5 pro**（OpenAI 兼容接口） | base_url `https://token-plan-cn.xiaomimimo.com/v1`；抽象 LLM 客户端，可一键换模型 |
| 技术栈 | **全 Node / TypeScript** | 本机无 Python、有 Node v22；docx 用纯 Node 解析已验证可行 |
| 形态 | Web 应用 | 前后端分离 |

> 注：测试稿 `test/AI替你炒股_紧凑版.docx`（37 段财经评论）= **待去 AI 味的目标稿**。

## 2. 系统架构

```
前端 (React + Vite)              后端 (Python + FastAPI)            本地模型
──────────────────               ───────────────────────           ─────────────
① 上传目标 docx + 范文      ──►   解析 docx → 段落结构
                                  范文 → 风格画像
② 触发"整篇改写"            ──►   逐段改写(流式) ────────────►   OpenAI 兼容 /chat
   左右对照编辑器           ◄──   返回按段对齐的改写文本
③ 点某句 → "换个说法"       ──►   单句生成 N 候选 ──────────►   OpenAI 兼容 /chat
④ 选择候选 / 手动编辑
⑤ 导出                      ──►   句子重组 → 生成新 docx → 下载
```

**核心设计原则**：不预生成全文每句的候选（慢且贵），整篇先改写一次，候选按需即时生成。

## 3. 技术栈

- **后端**：Python 3.11 + FastAPI + Uvicorn
  - `python-docx`：解析 / 生成 docx
  - 中文切句：正则（`。！？；…` + 引号/括号边界处理），MVP 自写，必要时引入 `harvesttext`
  - LLM 客户端：`openai` SDK（指向本地 `base_url`，如 `http://localhost:11434/v1`）
  - SSE 流式输出（整篇改写边改边显示）
- **前端**：React + Vite + TypeScript
  - 状态管理：Zustand（轻量）
  - 对照编辑器：自定义组件（左原文 / 右改写，逐句可点）
- **本地模型**：Ollama（默认）；推荐 `qwen2.5:14b` 或更大。中文改写小模型效果差，需用户硬件支撑（建议 ≥16GB 显存或量化版）。

## 4. 数据模型

```
Document
  id
  paragraphs: Paragraph[]

Paragraph
  id
  style: "heading1" | "heading2" | "normal" | "list"   # 段落级保真
  original_text
  rewritten_text
  sentences: Sentence[]

Sentence
  id
  original          # 原句
  rewritten         # 整篇改写后的版本（候选默认值）
  current           # 用户最终采用的文本
  candidates: str[] # 点击时即时生成，缓存

StyleProfile          # 由范文提取
  summary             # 模型总结的风格特征（句长/用词/语气）
  samples: str[]      # 若干范文片段，作 few-shot
```

## 5. 后端 API

| 方法 | 路径 | 作用 |
|------|------|------|
| POST | `/api/upload` | 上传目标 docx + 范文，返回 Document（段落结构）+ StyleProfile |
| POST | `/api/rewrite` | 整篇按段改写（SSE 流式返回） |
| POST | `/api/sentence/alternatives` | 入参：句子 + 上下文 + 风格；返回 N 个候选 |
| POST | `/api/export` | 入参：最终各句文本；返回新 docx 文件流 |
| GET  | `/api/health` | 检测本地模型连通性 |

## 6. 关键技术难点与对策

1. **"AI 味"要可操作化**
   - 把它拆成具体规则写进 prompt：套话过渡（"综上所述""总而言之"）、排比堆砌、形容词过度、句式过于整齐、"不仅…而且…"滥用、空泛总结句等。
   - **范文是最好的反 AI 味样本**：对齐范文的句长分布、用词与语气，比抽象规则更有效。

2. **段落对齐**：改写时要求模型**严格逐段返回**，用 JSON 数组或唯一分隔符包裹，避免模型合并/拆分段落导致错位。

3. **中文切句**：处理句末标点、引号内的标点（"他说："好。"" 不应在内部切开）、省略号、数字小数点。

4. **docx 格式保真**：MVP 只保 **段落级**结构（标题层级、正文、列表），放弃字符级格式（个别字加粗/变色）。覆盖绝大多数"文章"类文档。列为后续增强项。

5. **本地模型质量与速度**：小模型中文改写差、流式延迟高。`/api/health` 启动自检；配置里可调 `model` / `base_url` / `temperature`。

## 8. 分阶段里程碑

### M0 · 脚手架与连通性（地基）
- 建前后端目录、依赖、配置
- `/api/health` 打通本地模型（Ollama）调用
- **验收**：能用一句话调本地模型返回结果

### M1 · 后端核心管线（不含 UI）
- docx 解析 → 段落结构
- 中文切句
- 段落重组 → 导出 docx
- **验收**：脚本跑「上传 docx → 解析 → 原样导出」往返不丢段落

### M2 · LLM 改写能力
- 范文 → 风格画像
- 整篇按段改写（去 AI 味，流式）
- 单句生成 N 候选
- prompt 模板沉淀到 `prompts/`
- **验收**：给定文章+范文，命令行能产出风格对齐的改写全文，并对指定句返回 3 个候选

### M3 · 前端交互
- 上传页（目标 + 范文）
- 左右对照编辑器：逐句展示
- 点句 → 弹出候选 + 手动编辑框 → 选定回填
- 导出下载
- **验收**：完整走通「上传→改写→改几句→导出」

### M4 · 打磨
- SSE 流式体验、错误与超时处理、并发候选缓存
- AI 味规则调优、段落对齐健壮性
- （增强）字符级格式保真、批量句改写

## 9. 主要风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 本地模型中文改写质量不足 | 改写效果差 | 推荐 Qwen2.5-14B+；LLM 客户端可一键切云端 API |
| 改写时段落错位 | 对照混乱 | 强制 JSON/分隔符逐段返回 + 校验回退 |
| 长文档速度慢 | 等待久 | 分块 + 流式；候选按需生成 |
| docx 复杂格式 | 导出失真 | MVP 限段落级，复杂格式列后续 |
